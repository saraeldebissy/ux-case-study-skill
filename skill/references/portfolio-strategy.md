# Portfolio Strategy & Context Awareness

Read this when the user is writing multiple case studies, targeting a specific role/seniority,
or when a portfolio-level view would meaningfully improve the case study being written.

---

## When to Invoke Portfolio Strategy

Ask portfolio context questions if ANY of the following are true:
- User mentions they're building or updating their portfolio
- User says they're job searching or targeting a specific role/seniority
- User has already written or is planning multiple case studies
- The project being documented seems atypical (very short, cancelled, solo, no metrics)
  and framing it in portfolio context would strengthen it

If this is clearly a one-off documentation task (e.g. internal record-keeping), skip this.

---

## Portfolio Context Questions

Ask these as a natural addition to Phase 2, not as a separate interrogation:

**Role targeting**
- What type of role are you targeting? (IC designer, lead, manager, specialist — e.g. research,
  systems, growth)
- What seniority level? (mid, senior, staff, principal)
- Any specific companies or industry verticals you're focused on?

**Portfolio narrative**
- How many case studies are in your portfolio, including this one?
- What skills or strengths do your other case studies already demonstrate?
- Is there anything you feel your portfolio is currently missing?

---

## How to Use Portfolio Context in Writing

### Differentiation
Each case study should prove something different. Map the current project to a skill gap:

| If portfolio is missing… | Lean into this project's… |
|---|---|
| Strategic thinking | Problem framing, the "why this problem" section |
| Research depth | Discovery methods, how findings changed direction |
| Cross-functional influence | Stakeholder dynamics, decision-making process |
| Craft / execution | Visual decisions, interaction details, polish |
| Business impact | Outcomes, metrics, business rationale |
| Self-awareness | Reflection section, honest constraints |

### Seniority Calibration
Adjust framing based on target level:

**Mid-level**: Show craft + process. Prove you can execute well and learn fast.
- Lead with: *what you made* and *how you made decisions*
- Keep reflection honest but forward-looking

**Senior**: Show judgment + impact. Prove you navigate ambiguity and move metrics.
- Lead with: *why this problem* and *what you chose not to do*
- Quantify impact wherever possible; be specific about tradeoffs

**Staff/Principal**: Show systems thinking + organizational influence.
- Lead with: *how this fits a larger strategy*
- Include: stakeholder alignment, how this unblocked other teams, what it unlocked next

### Atypical Projects
Some projects don't fit the standard mold. Frame them proactively:

| Project type | Framing angle |
|---|---|
| Cancelled / shipped but killed | "What I'd have done next" + learnings from what was built |
| Solo, no research budget | Constraint-driven design story; speed + assumptions + validation |
| Very short sprint (< 2 weeks) | Lean into decision speed; what you prioritized and why |
| No measurable outcomes yet | Early signals + what's being tracked + why it matters |
| Heavily NDA-constrained | The thinking is the story; anonymize entities, keep the design narrative |

## Multi-Case-Study Consistency Note

If the user is writing multiple case studies in this session, maintain a log of:
- Skills/strengths already demonstrated in previous case studies this session
- What the next one should differentiate on

Mention this to the user before starting Phase 1 of a second or subsequent case study.
